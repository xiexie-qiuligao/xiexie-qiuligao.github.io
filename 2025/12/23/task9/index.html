信息泄露

  1. Git泄露，Git 是分布式版本控制系统，用于跟踪文件（尤其是代码）的修改历史，方便多人协作开发和版本回滚。就相当于可以随时存档自己的开发的东西，然后假如有遇到新开发的版本有问题，可以及时做到版本回滚。每个开发者本地都有完整的版本库，.git目录就储存了整个仓库的所有版本的数据。泄漏的原因，或者说本质，就是.git被部署到了公网的web服务器上。![[Pasted image 20251120181153.png]]
    git的工作分为三个核心区域![[Pasted image 20251120181327.png]]
    那么我进行攻击的方式就是通过这个.git，用GitHack工具遍历.git里面的文件，然后用各种查询对里面的文件来查看

使用工具
py -2 GitHack.py http://bulabulabula.ctfhub.com:10800/.git/
这里要说到做题里面的stash，很常用，在开发的时候stash（暂存） 是用于临时保存工作区和暂存区未提交修改的核心功能，简单来说就是把当前的代码修改先藏起来，让工作区回到干净状态，之后可以随时恢复这些修改。（那么这里面很大可能就有我们要的数据）![[Pasted image 20251120182432.png]]这是开发者的使用，那么作为进攻者（莫名兴奋），一般的顺序就是先去验证他有没有stash记录git stash list
然后恢复最近的一次git stash pop（也不一定，可以指定，但是靶场没有涉及到，就用到图里面的编号）
若git stash list无结果，但目标.git目录包含.git/objects，可尝试直接执行git stash apply，Git 会自动检测并恢复可用的 stash，，恢复 stash 后，若工作区无明显 flag 文件，用git diff查看 stash 恢复的修改内容（git diff会显示本次 stash 的具体代码变更）
这里还学到了grep查找grep [核心参数] “匹配规则” 搜索目录
![[Pasted image 20251120183308.png]]![[Pasted image 20251120183417.png]]
这些是一些常用的搜索参数，然后./就代表当前目录（参数可以叠加）。
还有通过log（提交的版本来找的，查看本地仓库的所有提交历史）![[Pasted image 20251120184155.png]]
这里是一个攻击实例git log –oneline
9e434d6 (HEAD -> master) delete flag.txt # 最新提交：删除了flag
610cd9b first commit: add flag.txt # 早期提交：添加了flag
跟踪flag.txt的提交历史（即使已被删除）
git log –follow flag.txt
输出该文件的所有修改记录，确认删除它的提交哈希
查看“删除flag”提交的具体修改内容（-p参数）
git log -p 9e434d6
直接显示该提交中删除的flag.txt内容，无需回滚版本

还有就是暂存区，index，用于存放开发者已执行git add但未执行git commit的修改，也就是还没有被改到版本但是已经保存了的。可能是开发者把本地的 “项目文件夹”（包含所有文件 + 隐藏的.git目录）直接上传到服务器，并且服务器允许访问隐藏目录（.git），也可能是开发者忘了部署到版本里面，此时flag.txt只在 “暂存区”（.git/index），没进入 “版本库”（.git/objects）![[Pasted image 20251120185227.png]]

  1. 备份文件，这个好理解，就是说一个粗心的程序员，再把自己的文件边写边备份的时候，一疏忽居然将自己的备份文件一起传到公网上面了，导致所有用户都可以直接访问。以下是各种备份文件的后缀（示例文件名）![[Pasted image 20251120191313.png]]
  2. vim缓存文件，vim 是 Linux/Unix 下的常用文本编辑器，编辑文件时会生成临时缓存文件，若异常退出（如断网、服务器宕机），缓存文件不会自动删除，若这些文件位于 Web 可访问目录，就会造成vim 缓存泄露。![[Snipaste_2025-11-18_19-23-03.png]]这个是ctfhub上面的题
    有几个类型
  • 一个是交换文件，是二进制文件，就是vim的实时缓存，当你用 vim index.php 打开文件开始编辑时，vim 会立刻在当前目目录生成交换文件；
    编辑过程中，交换文件会实时同步工作区的修改（比手动保存更频繁）；
    正常退出 vim（按 :wq 保存退出）时，vim 会自动删除交换文件；
    异常退出（按 Ctrl+Z 强制退出、服务器宕机、断网）时，交换文件不会删除，保留最后一次编辑的内容
    以 .swp/.swo/.swn为后缀，最后一个字母依次递减（按照字母表）直接用这个
    vim -r .index.php.swp来还原
  • 备份文件，是文本文件，可以直接cat打开，等于自己保存的文件，会以~结尾，而且会一直保留，不会像交换文件一样退出后会自动删除（这个默认配置默认可能关闭，但很多开发者会手动开启，CTF 题目中通常已开启）
  • 撤销文件（这个需要手动开启），撤销文件是 vim 记录「所有撤销操作轨迹」的文件，能恢复到编辑过程中的任意历史状态，.文件名.un~
    vim -u NONE（忽视默认配置） -c “set undofile | set undodir=./ | edit index.php”

4. svn泄露，这个有点像git，是一款经典的集中式版本控制系统，核心作用是跟踪文件（尤其是代码）的修改历史、管理多人协作开发，SVN 泄露能还原源码，本质是.svn目录存储了项目的完整元数据。![[Pasted image 20251120200159.png]]
成因：开发者部署时未删除.svn 目录，导致每个人都能访问到.svn目录。 或者是Apache/Nginx 默认未禁止访问以.开头的目录，也就是说莫默认可以访问到这些隐藏的目录，访问会返回 403，但仍能通过工具检测目录是否存在。
cd dvcs-ripper
perl rip-svn.pl -v -u http://目标地址/.svn/ 来得到svn目录，这个更详细一点![[Pasted image 20251120200833.png]]
5. DS_Store,.是 Mac OS 保存文件夹的自定义属性的隐藏文件。通过.DS_Store可以知道这个目录里面所有文件的清单,这个我归类到了备份文件里面，一样的道理。
6. HG泄露，和git svn一样的（只是查看方式类似），只是目录名字不同![[Pasted image 20251120201636.png]]
7. 总结一下，我觉得信息泄露最大的原因其实是开发者的疏忽（能这样说吗，毕竟大多数安全问题都是人的疏忽，但是事实却是这样）就比如git svn hg 这些的泄露，几乎都是因为没有对他们的危险目录进行删除，或者是控制.的输入，然后造成了我们可以利用这些目录。还有就是一般的备份文件，也是因为人把他自己的备份文件（不一定是自己，也有可能是编辑器或者是系统自动的，但是总结来就是因为人去把这些备份文件一起传到了公网上面，导致我们可以来访问到备份文件），又或者是ctfhub上面的最简单的几关（可能是故意而为之，但是这种情况也不是一定不发生）就是自己把重要的文件一起打包发上去了。

RCE

  1. 执行系统命令
    exec()，执行系统命令，默认仅返回命令输出的最后一行
    语法：exec(string $command, array $output = null, int $return_var = null): string|false
    $command：要执行的系统命令；
    $output：可选，数组引用，存储命令的所有输出行；
    $return_var：可选，整数引用，存储命令的退出状态码（0 表示执行成功，非 0 为失败）。
    特点：无直接输出（需手动打印$output），适合需要获取结构化输出的场景，不输出二进制数据，我在靶场里面没有用到过。

system(),执行系统命令，直接将命令输出打印到页面.

passthru()
作用：执行系统命令，直接将原始输出（包括二进制数据）打印到页面，无返回值。

shell_exec() 无直接输出，需手动打印,命令的完整输出字符串 无需处理完整输出的文本命令，有一个特殊的，反引号等价性：PHP 中，将命令用反引号包裹，效果与shell_exec()完全一致（如ls -l等价于shell_exec(‘ls -l’)）![[Pasted image 20251121132742.png]]
2. 总结RCE绕过

  • 文件包含，(1)采用 include 等文件包含函数，并且需要包含的文件路径是通过用户传输参数的方式引入；
    （2）用户能够控制包含文件的参数，被包含的文件可被当前页面访问

文件包含获取 webshell 的条件：
（1）攻击者需要知道文件存放的物理路径；
（2）对上传文件所在目录拥有可执行权限；
（3）存在文件包含漏洞；

  • php伪协议![[Pasted image 20251121133646.png]]

    这里就需要用到很多个过滤方式（就是找到相同的表达），慢慢积累的过程，后面还有学到我会马上记录到这里面。

还有过滤掉目录符号后，可以用管道符结合起来绕过
1.每个命令之间用;隔开 说明：各命令的执行给果，不会影响其它命令的执行。换句话说，各个命令都会执行，但不保证每个命令都执行成功。.每个命令之间用&&隔开说明：若前面的命令执行成功，才会去执行后面的命令。这样可以保证所有的命令执行完毕后，执行过程都是成功的。每个命令之间用||隔开说明：||是或的意思，只有前面的命令执行失败后才去执行下一条命令，直到执行成功一条命令为止。| 是管道符号。管道符号改变标准输入的源或者是标准输出的目的地。.& 是后台任务符号（一般在查看文件后面跟上base64）。 后台任务符号使shell在后台执行该任务，这样用户就可以立即得到一个提示符并继续其他工作。
